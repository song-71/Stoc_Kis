# Stoc_Kis — KIS OpenAPI 한국 주식 실시간 트레이딩 시스템

## 핵심 모듈 구조

| 파일 | 역할 |
|------|------|
| `ws_realtime_trading.py` | 메인 트레이딩 엔진 (WSS 실시간 체결, 매수/매도, 잔고관리) |
| `ws_realtime_tr_str1.py` | 매도 전략 모듈 (순수 함수, 상태 없음) |
| `kis_auth_llm.py` | KIS API 인증, 토큰관리, WSS 연결, `data_fetch()` |
| `kis_utils.py` | **중앙 유틸리티 모듈** — 아래 "모듈화 준칙" 참조 |
| `config.json` | V2 멀티계좌 설정 (`trade_enabled`, `exclude_cash`, `max_invest`) |

## 모듈화 준칙 — kis_utils.py 중심

**모든 공용 기능은 `kis_utils.py`에 집중한다.** 다른 파일(트레이딩, 시뮬레이션, 데이터 파이프라인 등)에서 공통으로 쓸 수 있는 함수는 kis_utils.py에 작성하고, 각 파일에서 import하여 사용한다. 개별 파일에 유틸리티 성격의 함수를 중복 작성하지 않는다.

### kis_utils.py 주요 기능 분류

**로깅**
- `LogManager(log_dir, log_name, run_tag, prefix_callback)` — 3-tier 출력 (screen / screen+file / screen+file+telegram)
- 메서드: `log_screen()`, `log()`, `log_tm()`, `log_raw()`, `log_file_only()`

**텔레그램 알림**
- 트레이딩 내부: `_notify(msg, tele=True)` (ws_realtime_trading.py 로컬)
- 범용: `LogManager.log_tm()` 또는 `telegMsg.py`

**데이터 조회**
- `stQueryDB(code)` — 1d OHLCV parquet 조회 → dict (date, open, high, low, close, volume, pdy_close, R_avr 등)
- DuckDB + S3 parquet 경로 기반

**KRX 종목 조회**
- `KRX_code(query)` → `(code, name, nxt, market, iscd_stat_cls_code)`
- `KRX_code_batch(symbols)` → `dict[symbol] = iscd_stat_cls_code`
- 종목상태: 00=정상, 57=관리종목, 59=단기과열, 58=상장폐지예정
- **주의**: KIS REST API의 `iscd_stat_cls_code`는 KRX와 다른 의미. 종목상태 판단은 반드시 KRX 데이터 사용

**가격 계산**
- `calc_limit_up_price(prev_close, market)` — 상한가 계산
- `calc_limit_down_price(prev_close, market)` — 하한가 계산
- `round_to_tick(price, market)` — 호가단위 반올림
- `price_minus_one_tick(price, market)` — 1틱 차감

**설정 관리**
- `load_config(config_path)` → V2 멀티계좌 dict
- `save_config(config)`, `resolve_account_config()`, `get_config_value()`

**기타 유틸**
- `is_holiday()` — 휴장일 체크
- `now_kst_str()` → `"YYMMDD_HHMMSS"`
- `bucket_for_symbol(symbol)` — SHA1 기반 버킷 배정
- `print_table()` — 한글 폭 보정 테이블 출력

### 새 공용 기능 추가 시 규칙

1. **kis_utils.py에 함수로 추가**하고 다른 파일에서 `from kis_utils import ...` 사용
2. 특정 파일에만 쓰이는 로직이라도, 향후 재사용 가능성이 있으면 kis_utils.py에 배치
3. 전략 판단 로직(매수/매도 조건)은 `ws_realtime_tr_str1.py`에 순수 함수로 작성
4. 인증/API 호출 관련은 `kis_auth_llm.py`에 추가

## 매도 전략 모듈 (ws_realtime_tr_str1.py)

순수 함수로 구성, 상태를 갖지 않음:
- `check_opening_call_auction_sell(antc_prce, prdy_ctrt)` — 08:29~09:00 예상체결 매도 판단
- `check_opening_call_auction_cancel(antc_prce, prdy_ctrt, buy_price)` — 예상체결 매도 취소
- `check_realtime_sell(bidp1, wghn_avrg, ma*, ...)` — 정규장 실시간 매도 (trailing stop, EMA deadcross)
- `vi_buy_strategy(...)` — VI 매수 조건
- `check_vi_sell(...)` — VI 매도 조건
- `calc_sell_pnl(...)` — 수수료/세금 포함 손익 계산

수수료 상수: `FEE_RATE_MARKET_BUY=0.0015`, `FEE_RATE_MARKET_SELL=0.0035`, `FEE_RATE_LIMIT_BUY=0.00015`, `FEE_RATE_LIMIT_SELL=0.002`

## 장 운영 시간대 (RunMode)

```
PREOPEN_WAIT → PREOPEN_EXP → REGULAR_REAL → CLOSE_EXP → CLOSE_REAL → OVERTIME → EXIT
```

## 시간대별 처리절차

| 시간 | 구간 | ord_dvsn | 동작 |
|------|------|----------|------|
| 08:30~08:40 | 장전 시간외종가 | 05 | 전일종가 매수 |
| 08:50~09:00 | 예상체결 | 00 | 구독, 매도 판단 |
| 09:00~15:20 | 정규장 | 00 | 실시간 체결, Str1 매도 |
| 15:19~15:30 | 종가매수 | 00 | 동시호가 매수 |
| 15:40~16:00 | 장후 시간외종가 | 06 | 시간외 종가매수 |
| 16:00~18:00 | 시간외 단일가 | 07 | 시간외 단일가 매매 |

주문구분: `ORD_DVSN_NO_PRICE = {"01", "05", "06"}` → 가격 불필요

## 주요 상수/설정

- `VI_TRADE_MODE`: `"off"` / `"test_mode"` (1주 고정) / `"run_mode"` (정상)
- `CLOSE_BUY = True` — 15:19 종가매수 활성화
- `SNAPSHOT_INTERVAL_SEC = 300` — 지표 스냅샷 간격
- 계좌: main(43444822), syw_2(63614390)
- `_iter_enabled_accounts(trade_only=True)` — `trade_enabled` 기준 계좌 필터링

## 코드 작성 준칙

1. **종목코드**: 항상 6자리 `zfill(6)`
2. **로그 prefix**: `ts_prefix()` → `[YYMMDD_HHMMSS_{LOG_ID}]` (TR=트레이딩, MO=모니터 등)
3. **텔레그램 시작 메시지**: 새 프로그램 작성 시 시작 알림은 반드시 아래 형태를 따를 것:
   ```python
   f"{ts_prefix()} {PROGRAM_NAME} => 프로그램 시작 메시지"
   ```
   예시: `[260330_082502_TR] ws_realtime_trading.py => 오늘은 개장일이므로 프로그램을 시작합니다.`
4. **텔레그램 알림**: `_notify(msg, tele=True)` 또는 `LogManager.log_tm()`
5. **텔레그램 메시지 필수 규칙**:
   - 모든 텔레그램 메시지에는 **반드시 `ts_prefix()` 또는 `[YYMMDD_HHMMSS]` 형식의 시간정보**가 포함되어야 함
   - 모든 텔레그램 메시지는 **자체 로그 파일에도 반드시 기록**해야 함 (텔레그램만 가고 로그에 없는 상황 금지)
6. **스레드 안전성**: 공유 dict 접근 시 Lock 사용
5. **config.json**: JSON `true/false` (소문자) ≠ Python `True/False`
6. **새 유틸 함수**: kis_utils.py에 추가, 개별 파일에 중복 작성 금지
7. **CSV 등 파일 저장 시 한글 인코딩**: 반드시 `encoding="utf-8-sig"` 사용 (Excel 호환 BOM 포함)
8. **위험 작업 사전 확인**: 트레이딩 시스템(ws_realtime_trading.py 등)에 영향을 줄 수 있는 변경(자동 재시작, 상태 초기화, WSS 연결 변경 등)은 반드시 사용자에게 위험성을 먼저 알리고 확인을 받은 후 실행할 것
9. **research 파일 날짜 표기**: `ws_monitoring_research_*.md` 등 research 파일 작성/수정 시, 메인 제목(h1) 또는 첫 줄 주석에 **작성일 및 수정일**을 반드시 명시 (예: `# WS Monitoring Research 260407 (작성: 260407, 수정: 260408)`)
10. **서버 오류 가설 금지 — 코드부터 의심**: 국내 증권거래 서버(KIS, KRX 등)는 매우 안정적이어서 서버 측 장애·점검·throttle 가 원인인 경우는 **거의 없다**. 연결 거부, 핸드셰이크 실패, 타임아웃, 응답 누락 등이 발생해도 **"서버 문제일 수 있다"는 가설을 먼저 제시하지 말 것**. 반드시 다음 순서로 자기 코드부터 다시 살핀다:
    1. 직전 세션의 좀비 thread / reconnect loop / 미정리 자원이 남아있는지
    2. 종료 처리(shutdown, close, stop_event 전파)가 누락된 지점이 있는지
    3. 동일 appkey/세션을 중복 사용하고 있는지
    4. 라이브러리 lifecycle 을 잘못 다루고 있는지 (kws.start/close 의 reconnect 동작 등)
    5. 위 모든 항목이 깨끗히 검증된 후에야 외부 원인을 거론할 수 있다
   "서버 점검 확인해보세요" 같은 회피성 답변을 사용자에게 하지 말 것.
11. **단정 금지 — 사실과 추정을 분리 표기**: "차단 아니다", "좀비다", "정상이다", "appkey 단위 throttle 이다" 같은 결론을 단정문으로 제시하기 전에 반드시:
    1. **사실** (직접 확인된 로그 라인·명령 출력·코드 라인) 과 **추정** (가설·해석) 을 명확히 구분 표기
    2. 추정에는 "추정", "가능성", "~로 보입니다" 등 가능성 표기를 명시
    3. 결론 단정 전 사용자가 준 식별자(IP, PID, 시각, 숫자, 메시지 본문)와 본인이 명령으로 얻은 식별자가 정확히 같은지 한 번 더 대조
   사용자가 정정해준 사실을 다음 답변에서 또 같은 단정으로 되풀이하면 안 됨. 단정 한 번 정정받으면 그 결론은 폐기.
12. **시간 표기 단위 명시 — UTC/KST 자동 매핑 금지**: `ps lstart`, `/proc` 타임스탬프, 파일 mtime, log 라인의 시각 등 모든 시간 정보는 그것이 **UTC 인지 KST 인지 매번 명시**한 후 변환할 것. "ps 표시 May05 = KST 5/5" 같은 무의식적 자동 매핑 금지. 한국 시장 트레이딩 환경 특성상 UTC↔KST 9시간 차이가 영업일/휴일 판정 자체를 뒤집을 수 있음.

| 등급 | 파일명 | 설명 |
|------|--------|------|
