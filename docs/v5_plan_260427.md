### F. Exit 통합 (기존 EMERGENCY_29PCT_SELL 대체)

```
max_prdy_ctrt ≥ 29.5% 도달 시:
```

**25% 하회 감지 시 (클라이언트 측 WSS 기반)**:

**29% 재상회 후 재매수**:

- 일일 한도 `UPLIMIT_MAX_DAILY_BUYS` 는 유지하지 않음 — 사용자 요청 "무제한"

**EMERGENCY_29PCT_SELL 통합**:
- 기존 `_check_sell_by_prdy_ctrt_periodic()` (:2070) 의 "29%+ 즉시 시장가 매도" 로직을 **위 단계청산 로직으로 대체**

### ② 13:55 재시작 후 수신건수 대폭 감소 (초당 수건)

**로그 측정**:
- 09:53 첫 재시작 후 정상 운영: 초당 100~500건/초 (30종목)
- 13:43 재시작 후: 초당 8~9건/초 (28종목) — **10배 이상 감소**
- 14:17 이후: 0건/초 (완전 끊김)

**가능한 원인 3가지**:

(A) **오후 시장 자연 감소** — 장 시작 직후(09:00~10:00) 대비 14시대는 본질적으로 거래 빈도 낮음. 그러나 일반적으로 100건/초 → 50건/초 수준이 정상. **8~9건/초는 비정상적으로 낮음**.

(B) **H0STMKO0 장운영정보 구독 슬롯 미사용** — a2 WSS (장운영정보 전용) 가 재시작 시 제대로 붙지 않아 일부 종목의 체결 스트림이 누락됐을 가능성.

(C) **재시작 시 approval_key 재발급 부분 실패** — 재시작 직후 일부 구독은 성공, 일부는 실패하여 28종목 중 실제 체결 수신되는 종목 수가 훨씬 적었을 수 있음. 로그상 "수신 종목 수 28개 = 구독 종목 수 28개" 라 되어 있지만 이게 "등록된 구독 종목 수" 이지 "실제 체결 수신된 종목 수" 가 아닐 수 있음.

**점검 방법** (내일 로그로 확인):
- `_code_last_recv_ts` 또는 `_recent_tick_ts` 종목별 마지막 수신 시각을 주기 로깅
- 재시작 직후 10분간 각 종목 수신 카운트 로깅

### ③ Watchdog 이 WSS 끊김 재시작을 못 한 이유

**현재 코드 watchdog 구조**:
- `_price_watchdog_loop` : 종목별 WSS 미수신 시 REST 보강 — **전체 WSS 재연결 안 함**
- `run_ws_forever` : 예외 시 재연결 루프 — **`auth_ws()` 실패 시 조용히 return** 하고 루프 계속
- `runner.sh` : 프로세스 `exit_code != 0` 시에만 재시작 — **프로세스가 안 죽으면 재시작 안 함**

**오늘 14:17 이후 실제 시퀀스**:
```
WSS 서버 close frame 없이 끊음 → websockets 예외 → run_ws_forever loop 다음 iteration
  → auth_ws() 호출 → "Get Approval token fail!" (KIS 서버 응답 문제)
  → ka._base_headers_ws["approval_key"] 갱신 안 됨 (빈 값 유지)
  → ka.KISWebSocket.subscribe() 호출해도 유효 키 없어 조용히 실패
  → 루프 계속 (프로세스는 살아있지만 수신 없음)
```

### Strategy C: 볼린저 밴드 squeeze → expansion + 반등 패턴 (사용자 제시)

**매수 조건 (AND)** — "변동성 압축 후 하단 이탈-복귀 반등":
