# 버전 비교: 11:30 재시작 vs 13:48 재시작 (2026-04-16)

## 요약 결론

**11:30 버전(4.1 원본 + P0 + P1)**이 **13:48 버전(v0416 + P1)**보다 훨씬 안정적.

- 11:30 버전: 2시간 10분 연속 무중단 (connect attempt=1만)
- 13:48 버전: **10분 만에 끊김** + "ALREADY IN USE appkey" 에러

v0416에서 추가된 기능(a2 분리, on_result fast-path) 중 하나가 WSS 안정성에 영향을 주는 것으로 강하게 의심됨.

## 의심 원인 분석

### 가설 1: a2 WSS 분리가 a1 안정성에 영향 (가능성 높음)

**근거:**
- 11:30 버전: a2 없음 → 2시간 10분 안정
- 13:48 버전: a2 연결 → 10분 만에 끊김
- "ALREADY IN USE appkey"는 a1 재연결 시 발생 (서버가 이전 a1 세션을 아직 정리 안 함)

**메커니즘 (추정):**
- a1 appkey ≠ a2 appkey (다른 계좌)이므로 공식상 2세션 허용
- 하지만 같은 PC/IP에서 두 WSS를 유지하면 서버가 이상 패턴으로 감지할 수 있음
- a2 쪽에서 간헐적 재연결/구독 변경이 a1 세션에도 간접 영향 가능

**KIS 공식 문서:**
> 1개의 PC에서 여러 개의 계좌(앱키)로 세션 연결 가능
> 계좌(앱키) 단위로 유량 제한(1개의 계좌(앱키)당 1세션)

→ 공식상 허용이지만 실제 운영에서는 단일 세션이 더 안정적일 가능성

### 가설 2: on_result fast-path가 WSS 콜백을 블로킹 (가능성 중)

**근거:**
- 13:48 버전은 매 WSS 틱마다 `_fast_29pct_sell_check()` 호출
- `partition_by` + `_str1_sell_state_lock` 획득 → 콜백 스레드 지연
- 초당 100건 이상 수신 시 콜백이 밀리면 ping/pong 응답 지연 → 서버가 세션 종료

**한계:**
- 오늘은 `_str1_sell_state`가 비어있어서 함수 초반에 return → 실제 부하 낮음
- 단, partition_by는 state가 비어있어도 실행됨

### 가설 3: 매도 즉시 스레드 (영향 없음)

**근거:**
- 오늘은 보유종목 0건 → 매도 로직 실행 안 됨
- 따라서 오늘의 WSS 끊김과 무관
