# WS Monitoring Research 260626: 코스피 서킷브레이커·사이드카 미인식 사건

> 2026-06-26 코스피 시장에서 매도 사이드카(11:12)와 1단계 서킷브레이커(12:10)가 실제로 발동했으나, 프로덕션(`ws_realtime_trading.py`)의 장운영정보 감지 로직이 이를 "서킷/사이드카 이벤트"로 인식하지 못한 사건에 대한 분석입니다. 코드 수정은 사용자 보류 지시에 따라 진행하지 않았으며, 분석 결과만 기록합니다.

## 실제 시장 사건 (뉴스 확정)

- **11:12:12 매도 사이드카**: 코스피200 선물이 전일 종가 대비 -5% 하락하여 프로그램매매 호가 효력이 정지되었고, 5분 후 자동 해제되었습니다.
- **12:10:12 1단계 서킷브레이커**: 코스피 지수가 전일 종가 대비 -8.19%(731.97p 하락, 8,198.33p)까지 떨어져 1분간 지속되어 발동되었습니다. 유가증권시장 전 종목 및 관련 선물·옵션이 20분간 매매거래 중단되었고, 12:30:12에 해제되었습니다.

## Issue 1: 서킷브레이커가 "거래정지(trht_yn=Y)" 1건으로만 인식됨 — 시장 전파 미발동

### 현상
```
# out/logs/wss_TR_260626.log:8497 (12:10:12)
[장운영정보] 다스코(058730) mkop_cls_code=AF8 antc_mkop_cls_code=112 trht_yn=Y

# out/logs/260626_telegram.log:113 (12:10:12)
[종목상태] 다스코(058730) 거래정지 (H0STMKO0, trht_yn=Y)
```
- 서킷브레이커 발동 순간, H0STMKO0(장운영정보)을 구독 중이던 코스피 종목은 다스코(058730) 단 1개뿐이었습니다.
- 그 1개 종목에서 `trht_yn=Y` 가지만 동작하여 "거래정지" 텔레그램 1건이 나갔을 뿐, 시장 전체 서킷브레이커로는 전혀 인식되지 못했습니다.

### 원인 (ws_realtime_trading.py:6572-6577, 8448-8463)
- 장운영정보 이벤트 감지 테이블은 서킷/사이드카를 **숫자 코드**로 기대합니다.
```python
# ws_realtime_trading.py:6572-6577
_MKOP_EVENT_NAMES = {
    "187": "사이드카발동", "388": "사이드카해제",
    "397": "사이드카매수발동", "398": "사이드카매수해제",
    "174": "서킷브레이크발동", "175": "서킷브레이크해제",
    "182": "서킷브레이크장종동시마감", "184": "서킷브레이크개시", "185": "서킷브레이크해제",
    "164": "시장임시정지",
}
```
- 그러나 오늘 실측된 값은 `mkop_cls_code=AF8`(발동) / `AF1`(해제), 보조필드 `antc_mkop_cls_code=112` 였습니다(wss_TR_260626.log:8497, 9873). 이 영문 코드들은 테이블에 전혀 없습니다.
- 감지 분기는 mkop → antc_mkop 순으로 테이블을 조회하는데(8448-8451), 둘 다 매칭되지 않아 `_evt_code` 가 빈 문자열로 남습니다.
```python
# ws_realtime_trading.py:8447-8452
_evt_code, _evt_src = "", ""
if _MKOP_EVENT_NAMES.get(mkop):
    _evt_code, _evt_src = mkop, "mkop_cls_code"
elif _MKOP_EVENT_NAMES.get(antc_mkop):
    _evt_code, _evt_src = antc_mkop, "antc_mkop_cls_code"
if _evt_code:   # ← 빈 문자열이므로 이 블록 전체가 통째로 미실행
    ...
    if _evt_code in ("174", "184", "164") and market:   # 8463 마켓전파 진입 조건
```
- 결과적으로 `[장운영]` 경고 로그(8458-8460), 텔레그램 알림, 마켓전파(8463-8479)가 모두 미발동했습니다.
- 참고: 174/184 가 실제 로그에 잡힌 적은 과거에도 0건입니다. 즉 KIS H0STMKO0는 처음부터 숫자 코드가 아니라 AF계열 영문 코드로 보내고 있었던 것으로 보입니다.
- 260611 작업 당시 주석(8434-8436)이 "단일가/서킷이 어느 필드/코드로 오는지 다음 발동 때 확정"하기 위해 원값 로깅을 추가해 두었고, 오늘 12:10 발동이 바로 그 표본 확보 시점이 되었습니다. **실측 결론: 정지=AF8, 해제=AF1, 보조필드 antc_mkop=112.**

### 미작동한 안전장치 (파급)
`_evt_code` 가 비어 마켓전파가 막히면서 다음 안전장치들이 전부 작동하지 않았습니다.
- 전 종목 예상체결(ccnl→exp) 전환 미발동 (8477-8479 `_trigger_ws_rebuild`)
- 무수신 감시 정지 미발동 (8474 안내문구의 동작)
- 신규 매수 차단 미발동 — `_is_cb_active()`(6560-6562)는 `_market_wide_event` 를 보는데, 이 dict 가 빈 상태로 남아 매수 가드(7178, 7928)가 풀려 있었습니다.
```python
# ws_realtime_trading.py:6560-6562
def _is_cb_active(code: str) -> bool:
    return bool(_market_wide_event.get(_code_market_map.get(code, "")))
```
- 30분 자동복귀 타이머 미설정 (8470 `_cb_resume_at[market]`, `CB_AUTO_RESUME_SEC=1800`)

### 수정 방안 (보류 중)
- (a) `_MKOP_EVENT_NAMES` 에 `AF8`(서킷발동)/`AF1`(해제) 매핑을 추가하거나, 시장 전 종목이 동시에 `trht_yn=Y` 로 들어오는 패턴을 시장 전파 트리거로 보강.
- 단, AF8/AF1·112 의 KIS 공식 의미(서킷 전용인지, 일반 거래정지·단일가와 코드를 공유하는지)를 먼저 확인해야 오탐을 막을 수 있습니다(후속 과제 참조).

## Issue 2: 서킷 해제(12:30)가 trht 가지가 아닌 REST 보충 경로로 우연히 해제됨

### 현상
```
# out/logs/wss_TR_260626.log:9873 (12:30:12)
[장운영정보] 다스코(058730) mkop_cls_code=AF1 antc_mkop_cls_code=112 trht_yn=N

# 12:30:03 [REST보충] 다스코 temp_stop_yn=N
```
- 12:30:12 해제 시점에 `trht_yn=N` 으로 정상화 신호가 들어왔습니다.

### 원인 (ws_realtime_trading.py:8405, 8787)
- 거래정지 감지(8405)는 `trht_yn == "Y"` 일 때 `_halted_codes` 에 **추가**하는 가지만 있고, `trht_yn == "N"` 으로 풀릴 때 `_halted_codes` 에서 **제거**하는 대칭 가지가 없습니다.
```python
# ws_realtime_trading.py:8405-8410
if trht_yn == "Y" and code not in _halted_codes:
    _halted_codes.add(code)
    _notify(... "거래정지 (H0STMKO0, trht_yn=Y)", tele=True)
# ↑ trht_yn == "N" 일 때 _halted_codes.discard(code) 하는 대칭 처리 없음
```
- 실제로 다스코가 `_halted_codes` 에서 빠진 것은 H0STMKO0의 trht 가지가 아니라, **REST 보충 경로**의 정상 분기였습니다.
```python
# ws_realtime_trading.py:8786-8790
# ── 정상: REST 보충만 수행 ──
_halted_codes.discard(code)
_enqueue_rest_price_row(code, output)
logger.info(f"{ts_prefix()} [REST보충] {name}({code}) 현재가={pr_fmt} temp_stop_yn={temp_stop}")
```
- 즉 해제는 의도된 H0STMKO0 처리가 아니라 REST 폴링이 우연히 종목을 다시 정상으로 잡아준 결과입니다. H0STMKO0 단독으로는 거래정지 해제 알림이 나가지 않습니다.

### 수정 방안 (보류 중)
- `trht_yn == "N"` 진입 시 `_halted_codes.discard(code)` + 해제 알림을 추가하는 대칭 처리.

## Issue 4: 무수신 워치독 — 이번엔 생존, 그러나 전체 정지 시 오발동 잠재 위험

### 현상
```
# out/logs/watchdog_260626.log:148-153 (12:10~12:35)
12:10:08 | [check] pids=[351453] status=1 ... 12:35:08 | [check] pids=[351453] status=1
```
- 서킷 구간 내내 프로세스는 status=1로 생존했습니다(하드종료 미발동).

### 원인 (ws_realtime_trading.py:5683, 5699-5713)
- 무수신 워치독은 종목별이 아니라 전역 `_last_any_recv_ts` 기준으로 판정합니다. 임계 180초 도달 시 `os._exit(2)` 합니다.
```python
# ws_realtime_trading.py:5683
WSS_NO_RECV_EXIT_SEC = 180.0     # 180초 무수신 시 os._exit(2)

# ws_realtime_trading.py:5699-5701
if (dtime(9, 0) <= _now_t < dtime(15, 20)
    and not _market_wide_event          # [260611] 서킷/시장정지 중엔 무수신 정상 → 하드종료 금지
    and (time.time() - _process_start_ts) > WSS_WATCHDOG_GRACE_SEC):
```
- 오늘은 코스닥이 계속 거래되어 전역 수신이 유지되었기에 워치독이 살아남았습니다.

### 잠재 위험 (미발동이지만 구조적 결함)
- 만약 코스피+코스닥 **전체**가 동시에 정지하는 더 큰 서킷이 왔다면, Issue 1의 미인식 때문에 `_market_wide_event` 가 빈 상태로 남습니다.
- 그러면 5700의 가드(`not _market_wide_event`)가 풀려, 180초 무수신 후 `os._exit(2)` 가 **오발동**할 위험이 있었습니다. 즉 이 가드는 서킷이 정상 인식되는 것을 전제로 하는데, Issue 1로 그 전제가 깨져 있습니다.

### 수정 방안 (보류 중)
- 무수신 워치독이 시장 정지를 서킷으로 오인하지 못해 하드종료하는 경우를 방지하는 보강(예: 시장 전 종목 동시 무수신 패턴 자체를 시장정지로 간주).

## Issue 5: 11:12 매도 사이드카 — 완전 미포착

### 현상
- 11:12:12 코스피200 선물 매도 사이드카 발동에 대해 프로덕션에 어떤 감지·알림 흔적도 없습니다.

### 원인 (정정 — 초기 오진 교정)
- **[정정] 초기 분석에서 "그 시각 코스피 H0STMKO0 구독 종목이 없었다"고 적었으나 이는 오진이었습니다.** keep-alive 타임라인 재구성 결과, 11:12 시점에 코스피·코스닥 모두 구독 중이었습니다:
  - 코스피: 다스코(058730) — 10:49 순환(`wss_TR:4591` `keep-alive 순환: 012205→058730 (KOSPI)`)으로 KOSPI keep-alive, 10:46 추가 후 12:10까지 해제 없이 유지.
  - 코스닥: 오에스피(368970) — 11:00 순환(`wss_TR:5067` `keep-alive 순환: 419050→368970 (KOSDAQ)`)으로 KOSDAQ keep-alive.
- **진짜 원인: 매도 사이드카는 개별 종목 H0STMKO0 장운영정보에 이벤트가 실리지 않는다.** 사이드카는 프로그램매매 호가효력정지일 뿐 개별 종목 매매상태(mkop_cls_code·antc_mkop·trht_yn)는 바뀌지 않습니다. 따라서 구독이 있어도 푸시할 사건이 없습니다.
  - 증거: `11:00~12:00` 사이 `[장운영정보]` 신호변화 로그 **0건**(다스코·오에스피 모두 그 5분간 H0STMKO0 푸시 없음). 원값 로거(`8438`)는 3개 필드 중 하나만 바뀌어도 남기므로, 신호가 왔다면 반드시 로깅됐을 것.
  - 11:10대 코스피 종목(광주신세계·금호건설·금호전기)은 정상 시세 계속 수신(`wss_TR:5483~5514`) → 개별 체결 지속 = 사이드카 특성 부합.
- 결론: 서킷브레이커는 전 종목을 **실제로 정지**시켜 구독 종목에 `trht_yn=Y`로 반드시 잡히지만(→ 12:10 다스코로 잡힘, 단 코드 미매핑), 사이드카는 종목 매매상태를 안 바꾸므로 현재 H0STMKO0 개별종목 레코드로는 **원리적으로 관측 불가**합니다.

### 영향 평가
- 설계상 사이드카는 정보성 처리(8461 주석: 프로그램매매 호가효력정지로 개인거래와 무관)라 운영 영향은 경미합니다.

### 후속 과제
- 사이드카는 H0STMKO0 개별종목 장운영정보로는 잡히지 않으므로, 감지하려면 시장지수·프로그램매매 전용 피드(또는 H0STMKO0의 우리가 읽지 않는 시장단위 필드 존재 여부 확인)가 필요합니다. keep-alive 구독 보장만으로는 해결되지 않습니다.
