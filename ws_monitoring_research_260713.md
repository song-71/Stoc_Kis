## 예상체결가가 안 들어온 메커니즘

1. VI 단일가(동시호가) 구간엔 체결이 성립하지 않으므로 KIS는 **실시간체결(H0STCNT0)을 보내지 않는다.** 대신 이 구간엔 **예상체결(H0STANC0)** 로 예상 체결가가 흐른다.
2. 그러나 H0STANC0 는 **해당 종목을 예상체결로 구독 중일 때만** 수신된다. 002880 은 09:02:00에 예상체결→실시간체결로 스왑되어 **H0STANC0 미구독** 상태였다.
3. 따라서 VI 구간에 예상체결가를 받으려면, 시스템이 **VI 발동을 감지하고 002880 을 실시간체결→예상체결로 되스왑(H0STANC0 재구독)** 해야 한다. 이 스왑이 안 일어났다 → 예상체결가 0건.

즉 문제의 본질은 "예상체결 채널이 죽었다"가 아니라 **"VI 발동을 못 잡아 예상체결 구독으로 되돌리지 못했다"** 이다.

## 추가 검토 — 09:00 예상체결 연장 로직과 슬롯 cap (사용자 질의 심화)

### (1) H0STMKO0 기반 조기 전환은 구현·활성돼 있다 — 002880엔 "작동 불가"였을 뿐

"VI 상태를 H0STMKO0로 확인하고 VI가 끝나면 즉시 실시간 전환"하는 로직은 코드에 살아 있고 켜져 있다: `VI_SWITCH_BY_VICLS = True` (`ws_realtime_trading.py:100`, 260701 도입). 전환은 **3단 구조**다.

| 단계 | 전환 트리거 | 코드 |
|---|---|---|
| ① 주력 | **H0STMKO0 vi_cls_code 'Y'→'N'(VI 공식 해제) 순간 즉시 전환** | `:8550` (`if VI_SWITCH_BY_VICLS and code in _vi_delayed_codes`) |
| ② 폴백 | 예상체결 `stck_oprc` 시가형성 감지 | `:12991` |
| ③ 폴백 | **09:02 고정 타이머** (`_vi_delay_until`) | `:6149` |

정의부 주석(`:96~99`)도 명시: *"True: H0STMKO0 vi_cls 'Y'→'N' 순간 즉시 전환(더 빠르고 정확), stck_oprc·09:02 타이머는 폴백."*

**그러나 002880은 ①의 전제인 H0STMKO0 구독이 안 돼 있었다**(아래 (2) 슬롯 cap 탈락). 주력 ①이 죽으니 결국 가장 거친 **③ 09:02 블라인드 타이머**로 전환됐다 — 로그 `09:02:00,184 VI 지연 해제 → 전 종목 실시간체결 전환`이 바로 ③이다. **VI 여부를 본 게 아니라 시각이 09:02가 돼서** 기계적으로 넘긴 것. 그래서 고등락 종목이 무방비 상태로 실시간에 놓였고, 2분 뒤 진짜 VI(09:04:48)를 맞았다.

→ **"의도한 로직이 반영 안 된 것"이 아니라, 슬롯 cap이 002880의 주력 경로(①)를 꺼버린 것**이 본질이다. 질문 1과 질문 2는 같은 근본 원인으로 수렴한다.

### (2) H0STMKO0 슬롯 cap(5)는 정적 상수 — 과보수적

- KIS WSS 등록 한도 = 41 → 상수 `MAX_WSS_SUBSCRIBE = 40` + 체결통보(H0STCNI0) 1슬롯 (`:832`).
- `H0STMKO0_MAX_SLOTS = 5` (`:8596`), 주석 *"데이터(ccnl/exp) 슬롯 ≥35 보장 위한 보수적 cap"*. **실제 데이터 구독 수와 무관하게 항상 5로 고정.**

**260713 실측 — 고정값의 낭비가 명확히 드러남:**
- 그날 데이터 구독은 **8종목뿐** (`[recv_분포] 구독=8`).
- 장초 총 사용 = `H0STCNI0=1, H0STMKO0=2, total=3/40` (`08:28:12` open_map) → **가용 슬롯 ~26개 여유.**
- 그런데도 08:59 관찰구독은 cap에 막힘: 보유종목 2개가 MKO0를 이미 점유 → `available = 5 − 2 = 3` → 후보 4개(한성·대구백·**디와이에이**·신테카) 중 **3개만 구독, 002880 탈락**.
- cap이 조금만 높았거나 동적이었어도 002880은 포함됐다.

**데이터 슬롯 보장은 이미 동적으로 지켜진다:** 데이터 구독 계산이 `available_slots = MAX_WSS_SUBSCRIBE − current_count − len(_mkstatus_sub_codes)` (`:10002`, `:13715`)로 MKO0 점유분을 차감한다. 따라서 **MKO0 cap을 상향/동적화해도 "데이터 ≥N 보장"은 깨지지 않는다.** cap을 고정 5로 둘 실질적 이유가 없다.

**개선 방향(제안, 미구현):** `H0STMKO0_MAX_SLOTS`를 고정 상수 → **동적**으로. 예: `가용 = MAX_WSS_SUBSCRIBE − 현재_데이터구독수 − 1(CNI0)` 범위에서 최소 보장치만 남기고 MKO0에 배분. 이러면 장초 고등락(VI 급등) 종목 전부가 H0STMKO0 관찰에 들어가 **①(주력 조기전환 경로)가 살아난다.**

## 핵심 결함 & 개선 방향(제안, 미구현)

두 결함이 **직렬로** 겹쳐 발생했다.

- **결함 1 (근본):** H0STMKO0 **슬롯 cap(5)이 정적 상수**(`ws_realtime_trading.py:8596`)라, 여유 슬롯이 ~26개 남아도는데도 고등락 종목(002880)이 관찰구독에서 탈락 → 의도된 주력 조기전환 경로 `VI_SWITCH_BY_VICLS`(①, `:8550`)가 002880엔 애초에 작동 불가 → 블라인드 09:02 타이머(③)로 넘어가 무방비 상태가 됨.
- **결함 2 (증폭):** REST "갑자기 끊김 → VI 의심" 감지의 **09:10 시작 게이트**(`:8951`). 개장 직후(09:00~09:10)에 상승 VI가 걸린, **이미 시가를 받은 종목**은 경로 B로도 잡히지 않는다.
- 두 결함이 겹쳐, 09:04:48 VI는 경로 A(cap 탈락)·경로 B(09:10 게이트) 모두 불발 → 예상체결 되스왑 실패 → 예상체결가 미수신.

**개선 방향(설계 필요):**
1. **H0STMKO0 cap 동적화** — `H0STMKO0_MAX_SLOTS` 고정 5 → `가용 = MAX_WSS_SUBSCRIBE − 현재_데이터구독수 − 1(CNI0)` 내에서 배분. 데이터 슬롯 보장은 `:10002`·`:13715`의 동적 차감이 이미 담당하므로 안전. 이러면 장초 고등락 종목 전부가 관찰에 들어가 ① 주력 경로가 살아남.
2. **REST 감시 개장초 확장** — 섹션 A/C 재정렬로 09:00~09:10 구간에도 "이미 시가를 받았지만 갑자기 끊긴 종목"에 `_avg_tick_interval` 기반 갑자기끊김 감지를 적용(09:10 게이트를 앞당기되 개장 초 REST 폭주 방지 조건 병행 설계).
※ 실제 구현·검증 방법은 로직 파일 §7(핸들러 리플레이) 참조. ①·② 중 하나만 고쳐도 이번 사건은 예방되나, 이중화를 위해 둘 다 권장.

- `_price_watchdog_loop` 09:00 가드 직후에 전용 감지 블록 신설(`:8864~`), 섹션(A) early-return 과 무관하게 **09:00~15:30 상시 동작**. 섹션(C)의 중복 블록은 제거(`:9004~`).
- 오탐 방지: `_avg_tick_interval`(데이터 부족 시 999.0)로 저유동·웜업 종목 자동 제외, 09:00~09:02 예상체결연장(`_vi_delayed_codes`) 종목 제외, `_vi_exp_sub_ts`·`_halted_codes` 제외, 종목별 재확인 10초 간격.
- 효과: 09:04 같은 개장 직후 VI 도 "활발히 거래되다 갑자기 멈춤"으로 감지 → REST 현재가조회 → `vi_cls='Y'` 면 경로 B 발동 → 예상체결 되스왑.

**이중 안전망:** ①은 경로 A(H0STMKO0)를, ②는 경로 B(REST)를 각각 개장 직후에 복원 — 둘 중 하나만 걸려도 예상체결 되스왑이 이뤄진다.

**로직 파일 갱신:** `rules/ws_realtime_trading_logic.md` §2·§6-1(신설)·변경이력(260714) 반영 완료.

**검증(미완):** 실장중 라이브 확인 필요(§7-C). 개장 직후 상승 VI 재현은 on-demand 불가하므로, 다음 장초 고등락 종목 발생 시 `[VI감지-REST]`가 09:10 이전에도 찍히는지 로그로 확인한다.
